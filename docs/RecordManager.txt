RecordManager (Менеджер записей)

Назначение:
RecordManager управляет записями фиксированной длины внутри файла таблицы,
используя страничную организацию, предоставляемую классом Pager. Он отвечает
за размещение записей на страницах, чтение, запись и добавление новых записей,
а также за сохранение метаданных таблицы (общего количества записей) для
обеспечения надёжности при перезапуске СУБД.

Основные структуры:
- Record - структура фиксированного размера (8 байт), состоящая из двух полей
  uint32_t: id (числовой идентификатор) и name_id (ссылка на строку в StringPool).
- total_records_ - текущее количество записей в таблице. Хранится как в
  оперативной памяти, так и на диске в метаданных страницы 0.
- RECORDS_PER_PAGE - константа, вычисляемая как PAGE_SIZE / sizeof(Record) = 512.
  Определяет, сколько записей помещается на одну страницу данных.
- Страница 0 - зарезервирована для хранения метаданных таблицы. В текущей
  реализации в её первых 4 байтах сохраняется значение total_records_.

Логика работы:

1. Инициализация
При создании RecordManager проверяется наличие страниц в файле через
pager_.get_num_pages():
- Если файл пуст (0 страниц), вызывается pager_.allocate_page() для создания
  страницы 0, в неё записывается total_records_ = 0.
- Если страницы уже существуют, из страницы 0 считывается сохранённое значение
  total_records_ (метод load_metadata()).
Таким образом, после перезапуска СУБД точное количество записей восстанавливается,
что предотвращает потерю данных при неполном заполнении последней страницы.

2. Адресация записей
Записи нумеруются последовательно, начиная с 0. Для нахождения физического
расположения записи с индексом N используется метод get_location():
  Номер страницы данных = 1 + (N / RECORDS_PER_PAGE)
  Смещение внутри страницы = (N % RECORDS_PER_PAGE) * sizeof(Record)
Единица добавляется, так как страница 0 занята метаданными.
Сложность доступа к произвольной записи - O(1).

3. Чтение записи (get_record)
- Вычисляется номер страницы и смещение через get_location().
- Страница целиком считывается с диска вызовом pager_.read_page().
- С помощью std::memcpy из страницы извлекаются 8 байт, формирующие структуру Record.
- Возвращается копия записи.

4. Запись (обновление) записи (write_record)
- Вычисляется расположение записи.
- Текущая версия страницы считывается с диска.
- По вычисленному смещению новые 8 байт копируются в страницу (через memcpy).
- Изменённая страница записывается обратно на диск через pager_.write_page().

5. Добавление новой записи (append_record)
- Индексом новой записи становится текущее значение total_records_.
- Вычисляется номер страницы, на которой должна располагаться эта запись. Если
  такой страницы ещё нет (выходит за пределы pager_.get_num_pages()), вызывается
  pager_.allocate_page() для расширения файла.
- Новая запись сохраняется вызовом write_record().
- total_records_ увеличивается на 1.
- Обновлённое значение total_records_ немедленно сохраняется на диск в страницу 0
  через save_metadata(). Это гарантирует согласованность метаданных даже при
  внезапном завершении программы после успешной записи данных.

6. Сохранение и загрузка метаданных
- save_metadata(): читает страницу 0, копирует текущее total_records_ в начало
  страницы (смещение META_OFFSET_TOTAL_RECORDS = 0) и записывает страницу обратно.
- load_metadata(): читает страницу 0 и восстанавливает total_records_ из первых
  4 байт.

Взаимодействие с другими модулями:
- Pager - предоставляет низкоуровневые операции чтения/записи страниц и выделения
  новых страниц. RecordManager использует только публичный интерфейс Pager, не
  обращаясь к файловой системе напрямую.
- StringPool (опосредованно) - хранит строковые значения, на которые ссылаются
  поля name_id в записях. RecordManager не работает со строками, а только с их
  идентификаторами.
- Table / Schema (в будущем) - будет использовать RecordManager для хранения
  строк таблицы, добавляя поддержку переменного числа колонок и типов.

Особенности реализации, важные для защиты:
- Страница 0 выделена под метаданные. Это стандартный подход, позволяющий
  расширять хранимую служебную информацию (версия формата, количество колонок,
  контрольные суммы) без изменения логики размещения данных.
- Атомарность обновления метаданных. После добавления записи данные и счётчик
  сохраняются на диск двумя последовательными операциями записи страниц. В
  случае сбоя между ними последняя запись может оказаться "потерянной"
  (неучтённой), но целостность существующих данных не нарушается. Для полной
  атомарности в будущем может быть добавлен Write-Ahead Log.
- Фиксированный размер записи. Позволяет вычислять положение любой записи за
  O(1) без обхода страниц, что критически важно для быстрого доступа по индексу
  (например, при использовании B-дерева).
- Надёжность. Благодаря сохранению точного количества записей в странице 0,
  перезапуск СУБД не приводит к потере доступа к части записей, даже если
  последняя страница данных заполнена не полностью.
